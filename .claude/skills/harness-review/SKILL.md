# Harness Review — 하네스 진단

3-strike로 누적된 위반의 **근본 원인을 하네스에서 찾는** 진단 스킬. `/retro`의 3-strike 감지 후 자동 호출되며, 진단 결과를 `/harness-update`에 전달한다.

## 사용법

`/retro`에서 자동 호출:

```
/harness-review SCOPE_CREEP
```

인자: 3-strike 누적된 위반 ID (예: `SCOPE_CREEP`, `LAYER_VIOLATION`)

---

## Phase 1: 위반 이력 수집

1. `retro_strikes.md`에서 해당 위반 ID의 모든 기록을 읽는다.
   - 날짜, 세션, 비고를 수집하여 **반복 패턴**을 파악한다.
2. 위반이 발생한 세션들의 공통점을 분석한다:
   - 같은 패키지에서 반복되는가?
   - 같은 유형의 작업(신규 구현, 리팩터링 등)에서 반복되는가?
   - 특정 상황에서만 발생하는가?

## Phase 2: 하네스 파일 탐색

위반 ID에 관련된 하네스 파일들을 탐색한다:

1. **CLAUDE.md** (루트) — 글로벌 지침에서 해당 위반 관련 규칙 확인
2. **`.claude/rules/*.md`** — 규칙 파일에서 해당 위반을 방지하는 규칙 존재 여부 확인
3. **`.claude/skills/*/SKILL.md`** — 관련 스킬에서 해당 위반 방지 조항 확인
4. **`.claude/settings.json`** — 관련 훅 존재 여부 확인

## Phase 3: 근본 원인 진단

위반이 반복되는 원인을 아래 카테고리로 분류한다:

| 원인 유형 | 설명 | 해결 방향 |
|----------|------|----------|
| **규칙 부재** | 해당 위반을 방지하는 규칙이 하네스에 없음 | 규칙 추가 |
| **규칙 모호** | 규칙은 있지만 해석이 불명확하여 에이전트가 다르게 해석 | 규칙 명확화 |
| **규칙 미노출** | 규칙은 있지만 해당 상황에서 로드되지 않음 (applyTo 누락) | 적용 범위 수정 |
| **검증 부재** | 규칙은 있지만 자동 검증 수단이 없음 | 훅 또는 체크 추가 |
| **스킬 누락** | 특정 워크플로우에서 검증 단계가 빠져 있음 | 스킬 절차 보강 |

## Phase 4: 개선안 도출

진단 결과를 바탕으로 구체적인 개선안을 도출한다:

1. **변경 대상 파일**: 어떤 하네스 파일을 수정해야 하는지
2. **변경 내용**: 추가/수정/삭제할 규칙 또는 절차
3. **기대 효과**: 이 변경으로 해당 위반이 방지되는 메커니즘

## Phase 5: 결과 보고 & `/harness-update` 트리거

진단 결과를 사용자에게 보고한다:

```
## 하네스 진단 결과

위반 ID: {VIOLATION_ID}
누적 횟수: 3회
반복 패턴: {패턴 요약}

### 근본 원인

{원인 유형}: {구체적 설명}

### 개선안

{변경 대상 파일과 변경 내용}
```

보고 후 `/harness-update`를 호출하여 개선안을 적용한다:

```
/harness-update {VIOLATION_ID} {진단 결과 요약}
```

---

## 규칙

- 이 스킬은 `/retro`의 3-strike 감지 후 **자동 호출**된다 — 독립 실행하지 않는다.
- 진단은 하네스 파일만 대상으로 한다 — 소스 코드 수정은 하지 않는다.
- 개선안은 최소 범위로 제안한다 — 하나의 위반에 대해 하나의 변경.
- `/harness-update` 호출 시 진단 결과를 인자로 전달하여 맥락을 유지한다.
