# Harness Update — 하네스 개선

`/harness-review`의 진단 결과를 받아, 하네스 파일(규칙/스킬/훅/설정)을 실제로 수정한다.

## 사용법

`/harness-review`에서 자동 호출:

```
/harness-update SCOPE_CREEP "규칙 모호: behavioral-guidelines.md의 범위 체크 규칙이 서브에이전트 작업에 적용되는지 불명확"
```

인자: `<VIOLATION_ID> <진단 결과 요약>`

---

## 핵심 원칙

| 원칙 | 요약 |
|------|------|
| **환경을 설계하라** | 에이전트 실패 → "환경에 무엇이 빠졌는가?" |
| **지도를 줘라, 매뉴얼 말고** | 거대 단일 문서 금지. 분산된 cross-linked docs |
| **단순화 = 성능 향상** | 도구/규칙 제거할수록 에이전트가 빨라지고 안정 |
| **Bitter Lesson** | 모델 업그레이드 시 불필요해진 규칙 제거 |
| **제약이 신뢰를 만든다** | 솔루션 공간 확장이 아니라 제약으로 품질 확보 |
| **토큰 = 비용** | 모든 규칙은 토큰 비용 정당화 필요 |

## Phase 1: 진단 결과 파싱

`/harness-review`에서 전달받은 인자를 파싱한다:

1. 위반 ID
2. 근본 원인 유형 (규칙 부재 / 규칙 모호 / 규칙 미노출 / 검증 부재 / 스킬 누락)
3. 개선안 요약

## Phase 2: 변경 위치 결정

근본 원인 유형에 따라 변경할 파일을 결정한다:

| 원인 유형 | 변경 대상 | 비고 |
|----------|----------|------|
| 규칙 부재 | `.claude/rules/{적합한 파일}.md` 또는 신규 생성 | 기존 파일에 추가할 수 있으면 기존 파일 우선 |
| 규칙 모호 | 해당 규칙이 있는 파일 | 규칙 문구를 명확하게 재작성 |
| 규칙 미노출 | 해당 규칙 파일의 frontmatter | `paths` glob 패턴 수정 |
| 검증 부재 | `.claude/settings.json` 또는 `.claude/rules/` | 훅 추가 또는 체크 규칙 추가 |
| 스킬 누락 | `.claude/skills/*/SKILL.md` | 해당 스킬의 절차에 검증 단계 추가 |

### 적용 범위 기준

- **글로벌**: `CLAUDE.md` — 모든 작업에 항상 적용되는 규칙만
- **디렉토리 스코프**: `*/CLAUDE.md` — 해당 디렉토리 작업 시에만 적용
- **규칙 파일**: `.claude/rules/*.md` — 특정 파일 패턴에 적용
- **스킬**: `.claude/skills/*/SKILL.md` — 특정 워크플로우에만 적용
- **설정**: `.claude/settings.json` — 훅, 권한 등 자동 실행 로직

## Phase 3: 변경 제안

변경 내용을 사용자에게 제시하고 `AskUserQuestion`으로 승인을 받는다:

```yaml
question: "아래 하네스 변경을 승인하시겠습니까?"
header: "하네스 변경 승인"
options:
  - label: "승인"
    description: "변경을 적용합니다"
  - label: "수정 요청"
    description: "변경의 특정 부분을 조정합니다 (notes에 기재)"
  - label: "거부"
    description: "변경을 적용하지 않습니다"
```

제안 시 아래 정보를 포함한다:

```
## 하네스 변경 제안

위반 ID: {VIOLATION_ID}
원인: {근본 원인 유형} — {설명}

### 변경 내용

파일: `{변경 대상 파일}`

{변경 전 → 변경 후 diff 또는 추가할 내용}

### 토큰 영향

- 변경 전: ~{N} tokens
- 변경 후: ~{M} tokens
- 차이: {+/-} tokens
```

- **승인 없이 하네스 파일을 수정하지 않는다.**

## Phase 4: 변경 적용

승인 후:

1. 하네스 파일을 수정한다.
2. 중복 규칙이 없는지 확인한다 — 같은 규칙이 여러 파일에 기록되지 않도록.
3. 변경된 파일의 토큰 수가 900 토큰 예산을 초과하지 않는지 확인한다.
   - 초과 시 파일 분할을 제안한다.

## Phase 5: 정합성 확인

변경 적용 후 아래를 검증한다:

1. **규칙 일관성**: 새 규칙이 기존 규칙과 모순되지 않는지 확인
2. **적용 범위**: 새 규칙의 applyTo가 의도한 범위만 커버하는지 확인
3. **토큰 예산**: 변경된 파일이 예산 이내인지 확인

검증 결과를 사용자에게 보고한다.

---

## 규칙

- 이 스킬은 `/harness-review`에서 **자동 호출**된다 — 독립 실행하지 않는다.
- 하네스 변경 전 반드시 사용자 승인을 받는다.
- 한 번에 하나의 위반에 대해서만 변경한다 — 범위를 넓히지 않는다.
- 규칙 추가 시 중복을 확인한다 — 같은 내용이 이미 다른 파일에 있으면 통합한다.
- 토큰 예산(파일당 900 토큰)을 준수한다.
- 변경 후 정합성 확인을 생략하지 않는다.
