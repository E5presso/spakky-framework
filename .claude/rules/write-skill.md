# 스킬 작성 규칙

## 하네스 수정의 스코프 원칙

이슈 처리 세션 중 발견된 **스킬/규칙/훅(하네스) 결함의 수정은 해당 이슈를 처리하는 워크트리에서 이뤄지며, 그 세션의 PR에 포함된다.** 루트 리포의 `develop` 브랜치에 직접 쓰지 않는다. 이 원칙은 이 규칙 파일 자체를 포함한 모든 하네스 파일(`.claude/skills/**`, `.claude/rules/**`, `.claude/settings*.json`)에 적용된다.

워크트리에서 하네스 파일을 수정할 때는 상대 경로(`.claude/skills/{name}/SKILL.md`)를 사용해 루트 오염을 방지한다 — 상세는 `.claude/skills/create-worktree/SKILL.md` 참조.

## 스크립트 코드 테스트 의무

스킬에 포함된 shell 스크립트, jq 필터, API 호출 등 실행 가능한 코드는 실제 환경에서 테스트한 후에만 커밋한다.

### 테스트 절차

1. 스킬에 포함된 각 스크립트 블록을 실제 PR/이슈에 대해 실행한다.
2. 기대 결과와 실제 결과를 비교한다.
3. 엣지 케이스(빈 결과, 동일 사용자가 여러 역할 수행 등)를 확인한다.

### 금지 사항

- 테스트하지 않은 jq 필터를 스킬에 포함하지 않는다.
- 스킬에 bash 스크립트 블록을 포함하려는 경우 해당 스크립트는 반드시 로컬 쉘에서 테스트 과정을 거쳐야 한다.
- 스킬 텍스트 내에 로컬 개발환경에 관련된 정보를 포함하지 않는다. (예: 특정 사용자 login, 프로젝트 ID 등)
- `gh --jq`의 제약(예: `--arg` 미지원)을 확인하지 않고 코드를 작성하지 않는다.
- 특정 사용자 login, 프로젝트 ID 등을 하드코딩하지 않는다. `gh api user --jq '.login'` 등으로 동적 취득한다.
