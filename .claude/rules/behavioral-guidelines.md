# 행동 원칙

Karpathy 4원칙 + 하네스 운영 원칙.

## 1. Think Before Coding

- 가정하지 말고, 불확실하면 물어본다.
- 해석이 여러 개면 전부 제시한다 — 조용히 하나만 골라 진행하지 않는다.
- 더 간단한 방법이 있으면 먼저 말한다.

## 2. Simplicity First

- 요청 범위를 넘는 기능을 추가하지 않는다.
- 단일 사용 코드에 추상화를 만들지 않는다 — 독립 헬퍼 함수 추출 금지. 2곳에서 같은 짧은 로직이 반복되면 인라인이 낫다.
- **소유권이 명확한 함수는 클래스에 귀속한다.** 특정 클래스에서만 호출되는 private 함수는 모듈 수준에 두지 않고 해당 클래스의 `@staticmethod`로 정의한다.
- **긴 orchestrator 메서드는 named phase로 분해한다.** 50줄 이상의 메서드가 여러 단계를 순차 실행하면, 각 단계를 이름 있는 staticmethod/메서드로 분리하여 진입점을 오케스트레이터로 유지한다.
- 요청되지 않은 "유연성"이나 "설정 가능성"을 넣지 않는다.
- 200줄을 50줄로 줄일 수 있으면 다시 쓴다.

## 3. Surgical Changes

- **PR close 절대 금지.** CI 재트리거를 위해 PR을 close/reopen하지 않는다. CI 실패 시 원인을 조사하고 코드를 수정하여 재push한다.
- **import와 사용 코드를 한 번에 추가한다.** import만 먼저 넣으면 linter(ruff)가 미사용으로 즉시 제거하여 추가→제거→재추가 루프가 발생한다. 새 함수/fixture를 추가할 때 해당 import를 같은 Edit/Write 호출에서 함께 작성한다.
- 꼭 필요한 것만 건드린다. 인접 코드, 주석, 포매팅을 "개선"하지 않는다.
- 망가지지 않은 것을 리팩터링하지 않는다.
- 기존 스타일에 맞추되, 내 변경으로 생긴 고아(orphan)만 제거한다.
- **범위 체크**: 변경된 모든 라인이 사용자 요청에 직접 연결되는가? 연결되지 않으면 되돌린다.
- **기존 기술을 교체/제거하기 전에 반드시 사용자에게 확인한다.** "X를 활용해줘"는 "Y를 제거해줘"가 아니다.
- **장애물을 만나면 우회하지 말고 근본 원인을 먼저 추적한다.** 안 되면 사용자에게 보고한다.
- **"이왕 하는 김에" 금지.** 버그 수정에 리팩터링을 끼우지 않는다. 별도 커밋/PR로 분리한다.

## 4. Goal-Driven Execution

- 작업 전 성공 기준을 정의하고, 검증될 때까지 반복한다.
- "버그 수정" → "재현 테스트 작성 후 통과시키기".
- "리팩터링" → "전후 테스트 통과 확인".
- 단계별로 `[단계] → 검증: [체크]` 형태로 진행한다.
- **스킬 Phase 전환 시 현재 Phase의 산출물이 존재하는지 확인한다.** 예: Phase 3 → Phase 4 전환 전에 워크트리가 생성되었는지 확인.

## 5. 서브에이전트 바이어스

**서브에이전트를 적극적으로 사용한다.** 서브에이전트는 fresh context에서 실행되므로 self-confirmation bias가 줄고, 메인 컨텍스트의 context rot을 방지한다.

- **검증/리뷰/회고 계열 작업**: 자기가 작성한 코드를 자기가 평가하면 편향이 생긴다. 서브에이전트가 독립된 시각으로 검토한다.
- **탐색/분석 작업**: 넓은 범위의 코드 탐색, 영향 분석 등은 서브에이전트로 수행하여 메인 컨텍스트를 보존한다.
- **병렬 가능 작업**: 독립된 패키지/파일에 대한 작업은 서브에이전트로 병렬 실행한다.

서브에이전트에게는 **완전한 컨텍스트**(패키지 경로, 실행 명령어, 기대 결과)를 제공하여 불필요한 추가 탐색을 방지한다.

## 사용자 질문 방식

코딩·계획·검수 등 맥락을 불문하고, 사용자에게 질문할 때는 다음을 따른다.

- **객관식을 우선 사용한다.** 선택지로 구성 가능한 질문에 자유 입력을 요구하지 않는다.
- **유력한 옵션에 `(권장)` 표시를 붙인다.** 판단에 소요되는 인지부하를 줄인다.
- **마지막에 항상 "Other (직접 입력)" 옵션을 둔다.** 예상 밖의 지시를 받을 수 있게 한다.

### 질의 자가검사 (질문 직전 의무 점검)

사용자에게 질의를 던지기 직전, 다음 3개 질문에 답한다. **셋 다 Yes면 질의 없이 자율 진행**한다. 하나라도 No면 질의 정당.

1. **답이 본문/사전/하네스에서 도출 가능한가?** (코드·CLAUDE.md·rules·메모리·티켓 본문 재독)
2. **권장안이 명백히 우월한가?** (트레이드오프가 미미하거나 한쪽이 압도적)
3. **권장안 근거를 1-2줄로 사용자에게 설명할 수 있는가?**

자가검사 결과는 진행 시작 메시지에 1줄로 노출 ("3-축 자가검사 통과: X 채택, 근거 Y").

질의 남발도 금지, 무분별한 자판정도 금지. **근거 기반 판단의 투명성**이 목표.

## Simplicity 구체적 시그널

`Simplicity First`(§2)는 다음 구체적 시그널로 점검한다. 1개라도 매치되면 단순화 검토.

- **단일 사용 헬퍼**: 호출자가 1개뿐인 private 헬퍼 → 인라인 또는 클래스 메서드로 귀속.
- **단일 사용 변수**: 1회만 사용되고 의미 추가가 없는 임시 변수 → 인라인.
- **Pass-through `__init__`**: 부모와 동일 시그니처를 그대로 전달하는 `__init__` → 삭제 (부모 사용).
- **Defensive None 체크**: 절대 None일 수 없는 자리에 `if x is None` → 삭제. 불변식이면 타입 시스템으로.
- **사용처 없는 "유연성" 매개변수**: 호출 시 항상 default 값만 받는 매개변수 → 삭제.
- **읽기 전용 wrapper 클래스**: 내부 객체 한 개를 그대로 노출하는 wrapper → 내부 객체 직접 사용.
- **Pre-mature Generic**: 타입 매개변수가 1개 구체 타입에만 인스턴스화되는 Generic → 구체 타입.

## 정보 우선순위

외부 스펙(GitHub Issue, 티켓 등)이 하네스 규칙 또는 현재 코드베이스와 충돌하면 **하네스와 코드베이스가 우선**한다. 티켓은 작성 시점의 가정을 반영하므로 현재 실정과 맞지 않을 수 있다. 우선순위 판단이 어려우면 사용자에게 확인한다.

## 하네스 운영

- **검증 후 문서화**: 하네스에는 검증된 사실만 반영. "아마도", "~일 수 있음" 형태의 규칙 추가 금지.
- **3회 실패 규칙**: 에이전트가 같은 실수를 3번 반복한 후에만 하네스에 규칙 추가.

## 세션 종료

- **PR 병합 완료 후 워크트리를 닫을 때** `/retro`를 **서브에이전트로** 실행한다. 자기 긍정 편향을 줄이기 위해 fresh context에서 독립적으로 평가해야 한다.
- 사용자가 세션 종료 의사를 밝히면(예: "끝", "마무리")에도 `/retro`를 **서브에이전트로** 실행한다.
