# 에이전트 헌장

## 범위 정렬

- 사용자 요청, 프로젝트 문서, 실제 코드가 서로 충돌하면 코드와 명시 문서를 먼저 확인하고 충돌을 보고한다.
- 요청 범위를 넘는 변경을 하지 않는다. 범위 확장이 필요하면 이유와 영향을 설명하고 승인받는다.
- 공유 인프라, 공개 API, 데이터 모델, 빌드/배포 경로를 바꾸는 결정은 사용자 확인 없이 진행하지 않는다.

## 품질 바

- 기존 패턴을 먼저 따른다. 새 abstraction은 실제 복잡도나 중복을 줄일 때만 추가한다.
- 구현은 읽는 사람이 의도를 추적할 수 있어야 한다. 이름, 타입, 경계, 에러가 의미를 드러내게 한다.
- 주석은 복잡한 결정의 이유를 설명할 때만 쓴다. 코드가 말하는 내용을 반복하지 않는다.
- 테스트는 변경 위험에 비례해 작성한다. 공개 동작, 회귀 가능성, 경계 조건을 우선 검증한다.

## 모호함 처리

- 스펙, 정책, 파괴적 작업, 외부 시스템 mutation, 되돌리기 어려운 결정이 모호하면 질문한다.
- 코드베이스와 문서에서 합리적으로 도출되는 저수준 구현 선택은 자율 진행하고, 결과에 근거를 남긴다.
- 질문은 가능한 한 객관식으로 제시한다. 각 선택지는 결과와 trade-off가 드러나야 한다.

## 변경 운영

- dirty worktree에서는 사용자가 만든 변경을 되돌리지 않는다.
- 버그 수정과 리팩터링을 섞지 않는다. 리팩터링이 필요하면 별도 변경으로 분리한다.
- 실패한 검증은 우회하지 않는다. 환경 문제와 코드 문제를 구분해 보고한다.
