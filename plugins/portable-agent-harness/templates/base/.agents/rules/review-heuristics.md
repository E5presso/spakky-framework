# 리뷰 휴리스틱

리뷰는 취향보다 운영 결함과 회귀 위험에 집중한다. 시그널이 구체적으로 매치될 때만 지적한다.

## P0/P1 후보

- 런타임 오류, 데이터 손실, 보안 취약점, 공개 API 호환성 파괴
- 레이어 의존 방향 위반, 플러그인/모듈 경계 위반
- 테스트로 잡히지 않는 의미 변경
- 문서가 공개 API, 환경변수, 설정, CLI 동작과 불일치

## 탐지 시그널

- 기존 아키텍처 패턴과 다른 구현이 들어왔지만 이유가 없다.
- 정적 분석이 닿지 않는 동적 코드가 늘었다.
- 계약이 타입, 인터페이스, schema가 아니라 런타임 관습에만 의존한다.
- `None`, 빈 값, 예외, fallback이 서로 다른 의미를 섞어 쓴다.
- 테스트가 실제 회귀를 검증하지 않고 snapshot이나 구현 세부만 고정한다.
- 시간, 순서, 네트워크, 외부 서비스에 의존하는 flaky 테스트가 추가됐다.

## 리뷰 방식

- 현재 변경이 만든 문제만 지적한다. 기존 문제가 새 변경으로 활성화되거나 악화된 경우는 예외다.
- 제안은 최소 수정 단위로 작성한다.
- 단순 네이밍 취향, 포맷터가 처리할 내용, 대안 선호만으로 코멘트를 남기지 않는다.
- 문제가 없으면 명확히 통과라고 말하고, 남은 검증 공백만 짧게 남긴다.
