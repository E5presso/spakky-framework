# 이슈 템플릿 & Stale 방지 원칙

본 문서의 템플릿은 **Phase 2.5 SDD 스펙 artifact의 슬롯**을 그대로 반영한다. 본문 섹션은 LLM이 재해석하지 않고 슬롯에 기계적으로 주입한다.

## 마일스톤 (에픽)

마일스톤의 description 필드에 **공용 스펙 문서** + **FR/SC 커버리지 매핑 표**를 주입한다. 자식 티켓은 이 description을 §번호로 참조하므로 description 내용이 축약되거나 생략되면 참조가 무효화된다.

```markdown
# {에픽 제목}

## §1. 비즈니스 의도

{이 마일스톤이 사용자/운영 관점에서 가져오는 변화 1-2 문단. 기술 어휘로 의도 대체 금지.}

## §2. 가정 (Assumptions)

- **사용자 컨텍스트**: {누가, 어떤 빈도로, 어떤 도구로}
- **데이터/환경**: {입력 데이터의 양·형태·정합성}
- **시스템 의존**: {다른 패키지·외부 API}
- **범위 가정**: {tenancy, locale, 권한 등}

## §3. 사용자 시나리오 (User Stories)

### US-1 (P1) — {제목}

**Story**: As a {actor}, I {capability}, so that {outcome}.
**Why this priority**: {비즈니스 가치·차단 관계·MVP 정의}
**Independent Test**: {단독 검증 가능 결과 1줄}

**Acceptance Scenarios**:
- **Scenario 1**: Given {초기}, When {액션}, Then {결과}.
- **Edge — {이름}**: Given ..., When ..., Then ...

### US-2 (P2) — ...

## §4. 기능 요구사항 (Functional Requirements)

- **FR-001**: {단일 명제}.
- **FR-002**: {단일 명제}.
- **FR-003**: ...

## §5. 도메인 계약 (Domain Contract)

- **입력**: {타입 + 의미 + 유효 범위}
- **출력**: {타입 + 의미 + 가능한 상태}
- **사전조건**: {호출자 보장}
- **사후조건**: {시스템 보장 결과 상태}
- **불변식**: {작업 전후 유지 조건}
- **상태 전이**: {필요 시 다이어그램 또는 표}

## §6. 도메인 규칙

- {비즈니스 불변식 1}
- {정책/계산 규칙 2}

## §7. 경계 조건 / Edge Cases

- {입력 0/empty/null}
- {동시 수정·중복 호출}
- {외부 의존 실패 시 재시도/fallback}

## §8. 상호작용

- 소비 이벤트/API: {이름 + 페이로드 의미 + ordering 가정}
- 발행 이벤트/API: {이름 + 페이로드 의미 + 발행 시점 + 멱등성}
- 외부 의존: {DB·메시징·외부 서비스}

## §9. 범위 밖 (Non-goals)

- {의도적 배제 1}
- {후속 티켓으로 이관 2}

## §10. 성공 기준 (Success Criteria)

- **SC-001**: {관찰 시점, 관찰 대상, 기대값}
- **SC-002**: ...

## §11. 아키텍처 변화 지도

```mermaid
{before → after 구조 다이어그램}
```

## §12. 용어 사전

- **{용어}**: {정의}. 적용 범위. 예시.
- **{신규 용어 X}**: 도메인 사전 등록 여부 + (등록되었다면) 의미 변화 diff + 재정의 비즈니스 이유.

## §13. FR 커버리지 매핑

| FR | 담당 자식 태스크 | 비고 |
|----|------------------|------|
| FR-001 | #{N} | |
| FR-002 | #{N}, #{N} | 후자는 검증 파트 |

## §14. SC 검증 절차

| SC | 기여 자식 태스크 | 검증 시점·도구 |
|----|------------------|----------------|
| SC-001 | #{N}, #{N} | 마일스톤 종료 후 운영 대시보드 |
```

자식 티켓은 이 문서를 "참조: 마일스톤 {이름} §{번호}"로 인용하며, 같은 내용을 자기 본문에 복사하지 않는다.

## 부모 이슈 (그룹 구조에서만)

부모 이슈는 자식 태스크들을 묶는 용도. `/process-ticket`의 소비 대상이 아니므로 분해 구조와 그룹 단위 스펙 요약을 담는다.

```markdown
# {그룹 제목}

## §1. 비즈니스 의도

{이 그룹이 해결하는 문제와 가져오는 변화. 1-2 문단.}

## §3. 사용자 시나리오 (요약)

### US-1 (P1) — {제목}
**Story**: As a {actor}, I {capability}, so that {outcome}.
- Scenario 1: Given ..., When ..., Then ...

(자식 태스크에서 시나리오 본문 인용)

## 작업 분해

| 자식 이슈 | 담당 FR | 선행 |
|-----------|---------|------|
| #{N} {제목} | FR-001 | -- |
| #{N} {제목} | FR-002, FR-003 | #{선행} |

## §10. 성공 기준

- **SC-001**: {관찰 시점, 관찰 대상, 기대값}

## §13. FR 커버리지 매핑

| FR | 담당 태스크 |
|----|-------------|
| FR-001 | #{N} |
| FR-002 | #{N} |

## 제약 사항

- {제약 1}
```

## 태스크 (기본 작업 단위)

**제목 형식**: `[{접두사}] {태스크명}` (그룹/에픽의 자식), 또는 접두사 없이 `{태스크명}` (단일)
- 예: `[Tracing] spakky-tracing 코어 패키지 생성`
- 순번을 붙이지 않는다. 개발 순서는 `blockedBy` 관계로 결정된다.

> `/process-ticket`이 소비하는 기본 단위. 본문이 user story·도메인 계약·FR/SC 역참조를 모두 포함하여 **상위 컨텍스트를 모르는 에이전트가 본문만으로 작업 가능**해야 한다.

```markdown
# {태스크명}

## 배경 및 동기

{이 태스크의 목적과 배경. 단일이면 자체 동기, 그룹/에픽이면 상위 목적 + 이 태스크의 역할.}

- **상위 목적**: {에픽/그룹의 비즈니스 의도 1줄} (단일이면 생략)
- **이 태스크의 역할**: {전체에서 담당하는 부분과 이유}

## 사용자 시나리오 (참조 US 본문 인용)

### US-{N} (P{1|2|3}) — {제목}

**Story**: As a {actor}, I {capability}, so that {outcome}.

**Acceptance Scenarios** (이 태스크가 통과시켜야 할 것):
- **Scenario 1**: Given {초기}, When {액션}, Then {결과}.
- **Edge — {이름}**: Given ..., When ..., Then ...

> 본문 인용. "US-1 참조"로만 두지 않는다 — Given/When/Then이 본문에 있어야 에이전트가 검증 단계로 전환 가능.

## 도메인 스펙

Phase 2.5 artifact §5/§6/§7/§8 중 이 태스크가 참조하는 섹션을 **그대로 인용**한다. 구현 디테일(파일 경로·시그니처)은 포함하지 않지만 **도메인 개념·규칙·계약·경계는 최대한 상세하게** 유지한다.

### 도메인 계약 (§5에서 인용)

- **입력**: ...
- **출력**: ...
- **사전조건**: ...
- **사후조건**: ...
- **불변식**: ...

### 도메인 규칙 (§6에서 인용)

- {규칙 1}

### 경계 조건 (§7에서 인용)

- {정상 vs 예외}

### 상호작용 (§8에서 인용)

- {다른 도메인/이벤트}

> 해당 없는 하위 섹션은 "해당 없음"으로 명시한다 (섹션 삭제 금지).

## 목표

{이 태스크가 달성해야 할 것을 1-2문장으로. 명확하고 검증 가능하게.}

## 선행 이슈

- [ ] #{선행번호}

> 선행 이슈가 없으면 이 섹션을 생략한다.
> 태스크 리스트(`- [ ] #N`) 형식으로 작성하면 GitHub이 자동으로 "Tracked by" 관계를 인식한다.

## 작업 범위

{수정/생성할 대상의 역할과 책임.}

- 대상 패키지: `{패키지명}`
- 참조할 기존 패턴: `{유사 구현 위치}의 {패턴명} 패턴 참조`

## §9. 범위 밖 (이 태스크 한정)

- {이 태스크에서 명시적으로 배제하는 항목}

## 수용 기준

**담당 FR**: FR-{NNN}, FR-{NNN}
**기여 SC**: SC-{NNN} (없으면 생략)

각 FR을 통과시키는 검증 가능한 항목:

- [ ] **FR-{NNN}**: Given {초기}, When {액션}, Then {결과} — {검증 방법: 단위 테스트 / 통합 테스트 / 수동 확인}.
- [ ] **FR-{NNN}**: ...
- [ ] §3 Acceptance Scenario 전부 자동 테스트로 통과
- [ ] lint + type check 통과

## 제약 사항

- {제약 1}
- {제약 2}

## 참고 컨텍스트

{에이전트가 작업 시작 시 읽어야 할 문서/코드 위치. 경로가 아닌 탐색 지시로 작성.}

- `ARCHITECTURE.md` 의존성 그래프 / 도메인 모델 섹션
- `AGENTS.md` "프로젝트 특수 컨벤션" (도메인 사전)
- `{패키지명}` 패키지의 기존 인터페이스 (ABC 기반 인터페이스)
- 관련 ADR (`docs/adr/`) — 문서 제목으로 참조
```

> **참고**: 그룹 구조에서 자식 태스크도 이 태스크 템플릿을 사용한다.

## 에픽 자식 태스크 (공용 스펙 참조 변형)

**에픽 구조(새 마일스톤 생성)에서만 사용.** 마일스톤 description에 공용 스펙 문서가 이미 존재하므로 자식 태스크 본문은 §5~§8을 참조로만 인용한다. 동일 내용을 복사하지 않는다 — 복사는 용어·계약 엇갈림의 시작.

단, **§3 사용자 시나리오는 본문에 인용한다** — Given/When/Then을 자식 본문에 두어야 에이전트가 검증 단계로 진입 가능.

```markdown
# {태스크명}

## 배경 및 동기

- **상위 목적**: 마일스톤 `{마일스톤 이름}` §1 참조
- **이 태스크의 역할**: {전체에서 담당하는 부분}

## 사용자 시나리오 (US 본문 인용)

### US-{N} (P{1|2|3}) — {제목}

**Story**: As a {actor}, I {capability}, so that {outcome}.

**Acceptance Scenarios**:
- **Scenario 1**: Given ..., When ..., Then ...
- **Edge — {이름}**: Given ..., When ..., Then ...

## 도메인 스펙

참조: 마일스톤 `{마일스톤 이름}` description §5 (계약), §6 (규칙), §7 (경계), §8 (상호작용).

추가 세부 (이 태스크 범위만 해당, 없으면 "추가 세부 없음" 명시):

- {이 태스크에만 국한된 추가 규칙·경계·예외}

## 목표

{이 태스크가 달성해야 할 것을 1-2문장으로.}

## 선행 이슈

- [ ] #{선행번호}

## 작업 범위

- 대상 패키지: `{패키지명}`
- 참조할 기존 패턴: `{유사 구현}의 {패턴명}`

## §9. 범위 밖 (이 태스크 한정)

- {이 태스크에서 명시적으로 배제하는 항목, 없으면 생략}

## 수용 기준

**담당 FR**: FR-{NNN}, FR-{NNN}
**기여 SC**: SC-{NNN}

- [ ] **FR-{NNN}**: Given ..., When ..., Then ... — {검증 방법}.
- [ ] **FR-{NNN}**: ...
- [ ] §3 Acceptance Scenario 전부 자동 테스트로 통과
- [ ] lint + type check 통과

## 제약 사항

- 공용 스펙의 제약을 따른다 (마일스톤 description §5~§8 참조).
- {이 태스크에만 적용되는 추가 제약, 없으면 생략}

## 참고 컨텍스트

- 마일스톤 `{마일스톤 이름}` description (공용 스펙 + FR/SC 매핑)
- {추가로 필요한 ARCHITECTURE.md 섹션·기존 패턴 위치}
```

> 자식이 본문에 유지하는 고유 내용: §3 US 본문 인용, "추가 세부", 담당 FR/기여 SC, 작업 범위.
> §5~§8 복사는 charter §1 "용어 엇갈림 방지" 위반.

## Stale 방지 원칙

이슈 본문에 **구현 디테일**(파일 경로·시그니처·커밋 해시·버전)을 넣지 않는다. 패키지/모듈 수준 지칭, 패턴 레벨 "참조할 기존 패턴" 안내, 탐색 지시어("`ARCHITECTURE.md` 의존성 그래프 섹션 참조")만 사용한다.

### 예외: 추상 스펙 밀도는 최대로

**도메인 개념·규칙·경계·계약**은 최대한 상세하게. 스펙이 빈약하면 `/process-ticket`이 할루시네이션으로 채운다.

- 금지: "`HealthCheckController.check(self) -> HealthStatus`를 구현한다"
- 허용: "HealthCheck는 등록된 Probe 0~N개의 결과를 집계하되, 한 Probe라도 unhealthy이면 전체를 unhealthy로 보고한다. Probe 등록은 DI Container 부트스트랩 시점에 결정된다."

### 예외: 사용자 시나리오는 본문 인용

§3 US와 Acceptance Scenarios는 자식 태스크 본문에 **그대로 인용**한다. "참조" 링크만 두면 에이전트가 검증 단계로 전환 시 별도 fetch가 필요해 누락 위험.

## 선후행 관계 표현

다음 메커니즘을 조합한다:

1. **Blocked by (Relationships UI)**: GraphQL `addBlockedBy` mutation으로 선행 이슈 관계를 설정한다. GitHub Relationships UI에 "Blocked by / Blocking" 관계가 표시된다.
   ```bash
   # 이슈 node ID 조회
   gh api graphql -f query='{ repository(owner: "{owner}", name: "{repo}") { issue(number: {N}) { id } } }' --jq '.data.repository.issue.id'

   # blocked by 관계 설정: {ISSUE}가 {BLOCKING_ISSUE}에 의해 블록됨
   gh api graphql -f query='mutation { addBlockedBy(input: { issueId: "{ISSUE_ID}", blockingIssueId: "{BLOCKING_ISSUE_ID}" }) { issue { number } } }'
   ```
2. **이슈 본문 메타데이터**: 이슈 본문에도 `## 선행 이슈` 섹션에 태스크 리스트(`- [ ] #N`)로 선행 이슈를 기재한다 (가독성 + `/process-ticket` 파싱용).
3. **마일스톤**: 태스크 이슈를 에픽 마일스톤에 연결하여 진행률 추적.
4. **GitHub Sub-issues**: GraphQL `addSubIssue` mutation으로 부모-자식 위계 설정 (그룹/에픽의 부모 이슈가 있는 경우).
5. **크리티컬 패스 & 병렬 그룹**: Phase 5 결과 보고에 명시.
