# Investigate — 버그 조사 워크플로우

증상 설명 또는 GitHub Issue 번호를 받아, 재현 → 원인 격리 → 수정 후보 제시까지 체계적으로 디버깅한다.

## 사용법

```
/investigate 42
/investigate "spakky-event에서 EventHandler 등록 시 중복 호출 발생"
```

인자: GitHub Issue 번호 또는 증상 설명 (자연어)

---

## Phase 1: 증상 수집

### 1-1. 입력 분류

- **숫자**이면 GitHub Issue로 간주:
  ```bash
  gh issue view $ARGUMENTS --comments
  ```
- **자연어**이면 증상 설명으로 간주하고 바로 Phase 2로 진행.

### 1-2. 증상 정리

수집된 정보를 아래 형식으로 정리한다:

- **증상**: 무엇이 잘못되었는가 (기대 동작 vs 실제 동작)
- **재현 조건**: 어떤 상황에서 발생하는가 (알려진 경우)
- **영향 범위**: 어떤 패키지/모듈이 관련되는가
- **에러 메시지/스택 트레이스**: 있으면 그대로 기록

## Phase 2: 재현

### 2-1. 관련 코드 탐색

**Explore 서브에이전트**로 증상과 관련된 코드를 탐색한다:

1. 에러 메시지, 클래스명, 함수명 등 키워드로 코드 검색
2. 관련 테스트 파일 존재 여부 확인
3. 최근 관련 커밋 확인: `git log --oneline -20 -- <관련 경로>`

### 2-2. 재현 테스트 작성

증상을 재현하는 **실패하는 테스트**를 작성한다:

1. 관련 패키지의 기존 테스트 구조를 확인한다.
2. 증상을 정확히 재현하는 최소한의 테스트를 작성한다.
3. 테스트가 **실패하는지 확인**한다:
   ```bash
   cd <package-dir> && uv run pytest <test-file>::<test-name> -x -v
   ```

재현 테스트 작성이 불가능한 경우 (환경 의존, 타이밍 이슈 등):
- 사유를 기록하고 Phase 3의 정적 분석으로 대체한다.

## Phase 3: 원인 격리

### 3-1. 가설 수립

증상과 코드를 기반으로 **가설 목록**을 작성한다. 각 가설은:

- **가설**: 원인으로 추정되는 것
- **검증 방법**: 어떻게 확인할 수 있는가
- **가능성**: 높음/중간/낮음

가능성이 높은 순서로 정렬한다.

### 3-2. 가설 검증

각 가설을 순서대로 검증한다:

1. **코드 분석**: 관련 코드를 읽고 로직 흐름을 추적한다.
2. **git blame/log**: 관련 코드의 변경 이력을 확인한다:
   ```bash
   git log --oneline -10 -- <파일>
   git blame -L <시작>,<끝> <파일>
   ```
3. **테스트 실행**: 기존 테스트로 가설의 일부를 검증한다.
4. **정적 분석**: 타입 체크, 의존성 방향 등 도구 기반 검증:
   ```bash
   cd <package-dir> && uv run pyrefly check
   ```

### 3-3. 근본 원인 확정

가설 검증 결과를 종합하여 근본 원인을 확정한다:

- **근본 원인**: 무엇이 문제인가 (코드 위치 + 로직 설명)
- **발생 경위**: 왜 이 버그가 생겼는가 (최근 변경, 설계 누락 등)
- **검증 증거**: 가설을 확정한 근거

## Phase 4: 수정 후보 제시

### 4-1. 수정 방안 도출

근본 원인에 대해 **1-3개의 수정 방안**을 도출한다. 각 방안은:

- **방안**: 구체적인 수정 내용
- **변경 범위**: 수정할 파일/함수 목록
- **장점/단점**: 트레이드오프
- **영향 패키지**: 다른 패키지에 미치는 영향

### 4-2. 결과 보고

아래 형식의 결과를 **반드시 텍스트로 화면에 출력**한다 (도구 호출이 아닌 사용자에게 보이는 텍스트 응답으로):

```
## 조사 결과

### 증상
{증상 요약}

### 근본 원인
{근본 원인 설명 — 코드 위치 포함}

### 수정 후보

| # | 방안 | 변경 범위 | 영향 |
|---|------|----------|------|
| 1 | {방안 1} | {파일 목록} | {영향} |
| 2 | {방안 2} | {파일 목록} | {영향} |

### 재현 테스트
{작성한 테스트 파일 경로, 또는 "재현 테스트 불가 — 사유"}
```

### 4-3. 다음 액션 제안

`AskUserQuestion`으로 다음 행동을 묻는다:

```yaml
question: "어떻게 진행할까요?"
header: "조사 완료"
options:
  - label: "수정 진행"
    description: "선택한 방안으로 수정을 시작합니다 (notes에 방안 번호 기재)"
  - label: "이슈 생성"
    description: "조사 결과를 GitHub Issue로 생성합니다"
  - label: "추가 조사"
    description: "특정 부분을 더 깊이 조사합니다 (notes에 기재)"
  - label: "종료"
    description: "조사 결과만 확인하고 종료합니다"
```

- "수정 진행" 선택 시 해당 방안으로 코드를 수정하고 `/check`로 검증한다.
- "이슈 생성" 선택 시 조사 결과를 이슈 본문으로 작성하여 GitHub Issue를 생성한다.

---

## 규칙

- 재현 테스트를 먼저 작성하고, 수정 후 해당 테스트가 통과하는지 확인한다.
- 가설 없이 코드를 수정하지 않는다 — 반드시 원인을 격리한 후 수정한다.
- 코드 탐색은 **Explore 서브에이전트**로 실행하여 메인 컨텍스트를 보존한다.
- 수정 시 범위를 최소화한다 — 버그 수정에 리팩터링을 섞지 않는다.
- `uv run` 접두사 필수.
- **루트에서 직접 실행 금지** — 반드시 패키지 디렉토리 내에서 실행.
