# 하네스 작성

하네스는 행동 원리의 모음이다. 문서가 아니라 규칙. 작성 시 아래 원칙을 따른다.

## 보편 원리만 작성

- 규칙은 일반화 가능한 행동 원리로 표현한다. "X 스킬의 Y 케이스 대응", "Z 작업의 A 동작" 같은 specific 인스턴스를 규칙으로 박지 않는다.
- specific 인스턴스가 누적되면 (a) 컨텍스트 윈도우를 잠식하고 (b) 범주 밖 케이스에 둔감해진다.
- 도메인 인스턴스(특정 도메인 인스턴스·collection·entity)는 규칙 본문이 아니라 도메인 사전·MEMORY·후속 이슈에 배치한다.

## 역사 기록 박제 금지 (MUST)

**회귀 사례·인시던트 로그·이슈 번호 인용·"본 세션의 회귀"·"사용자 지적으로 발견" 류 서술을 하네스에 박제하지 않는다.** 보편 원리가 본문에 이미 진술되어 있으면 인스턴스 부록은 행동을 변화시키지 않는다 — 토큰만 누적된다. 이 패턴은 본 하네스에서 가장 흔히 누출되는 안티패턴이며, 작성·리뷰 시 명시적으로 차단한다.

### 금지되는 형태

- 표 컬럼에 `회귀 시나리오 | #1078·#1080·#1083·...` 류 이슈 번호 나열
- `## 회귀 시나리오 (#N)` / `### 회귀 테스트 시나리오` 부록 섹션
- 본문 끝의 "회귀 사례: #N, #M ..." 인라인 인용
- "본 가드의 도입 사례", "사용자가 직접 X로 닫음", "#N 회귀의 진입점", "본 세션 wave 진행 중 ... 회귀가 관찰되었다" 류 역사 서술
- "Phase X 도입 사례", "#N 회귀의 한 축" 류 정당화 인용
- AC ID 인용으로 가드 정당화 (`#N AC #N`)

### 어디에 두는가

- **PR 본문 / commit message** — 인시던트가 일어난 시점에 변경의 정당화로 작성. `git log -S`·`gh pr list`로 사후 추적.
- **MEMORY** — 사용자 컨텍스트(선호·역할). 행동 규칙 박제용 아님.
- **하네스 본문** — 일반화된 시그널만. 인스턴스 트리거를 박제하지 않는다.

### 합법 인용 (예외)

ADR (Architectural Decision Record) ID는 **결정 근거**로 1줄 인용 허용 (예: "ADR ADR-520 §4 동격"). 회귀 사례 인용과 구분된다 — 결정 근거 그 자체가 보편 원리(설계 정당화)를 담고 있다. 이 경우에도 1줄·1회로 제한하며 "회귀 시나리오" 부록 섹션 형태는 금지.

## 검증 후 작성

- 검증된 사실만 반영한다. "아마도" 규칙 금지.
- 규칙 추가 후에는 `/evaluate-harness`로 차단력을 확인한다.

## 워크트리 격리 후 작성 (MUST)

하네스 파일(`.agents/rules/*.md`, `.agents/skills/**/*.md`, `AGENTS.md`)을 수정하기 전에는 현재 cwd가 작업 전용 git worktree인지 확인한다.

- `git rev-parse --show-toplevel`와 `git branch --show-current`를 확인한다.
- top-level이 메인 체크아웃(`.../spakky-framework`)이고 branch가 `develop`이면 하네스 수정을 시작하지 않는다.
- 루트/develop에서 변경 필요를 발견하면 먼저 `.claude/worktrees/<작업명>` worktree와 전용 branch를 만들고, 모든 Edit/Write/apply_patch/Bash mutation은 그 worktree 절대경로에서만 수행한다.
- 실수로 루트/develop에 tracked 변경을 만든 경우 즉시 변경분을 새 worktree로 옮기고, 루트에는 본인이 만든 tracked 변경만 되돌린다. 사용자·다른 에이전트의 untracked/무관 변경은 건드리지 않는다.

## 5가지 테스트 (모든 추가·기존 규칙)

1. **삭제 테스트**: 이 줄이 없으면 에이전트가 실수하는가?
2. **비추론 테스트**: 코드·도메인 사전으로 알아낼 수 있는가?
3. **중복 테스트**: 동일 규칙이 다른 파일에 있는가? SSOT (Single Source of Truth) 하나만 유지.
4. **행동 테스트**: 이 규칙 추가 후 에이전트 행동이 실제로 변하는가?
5. **보편 원리 테스트**: specific 인스턴스가 아니라 일반화 가능한 행동 원리로 표현되었는가?

테스트 1개라도 미통과면 추가 보류·기존 항목 삭제·축약. 토큰 = 비용 — 같은 차단력을 더 적은 토큰으로 얻을 수 있으면 그쪽이 default.
