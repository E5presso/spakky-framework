# 하네스 작성 규칙

`.agents/harness/`·`.claude/`·`.codex/`·`CLAUDE.md`·`AGENTS.md`·`RTK.md` 등 **하네스 자체에 추가·수정되는 모든 텍스트**에 적용된다.

## 1. 보편 원리만 작성

- 특정 스킬·티켓·인시던트의 인스턴스를 본문에 박제하지 않는다.
- "DOC-1234 회귀 사례", "PR #135에서 발생한 문제" 같은 부록 금지. 그런 사실은 PR 본문·커밋 메시지·ADR에 둔다.
- 규칙은 **재발 방지의 일반 원리**여야 한다. 단발성 사건의 기록이 아니다.

## 2. 5-Test 게이트 (규칙 추가·유지의 통과 기준)

신규 규칙 추가, 기존 규칙 유지·수정 시 다음 5개 테스트를 **모두 통과**해야 한다. 하나라도 실패하면 추가 거부 또는 삭제 후보.

### Test 1: 삭제 테스트
- "이 규칙을 지금 삭제하면 무슨 일이 벌어지는가?"
- 답이 "아무 일도 안 일어난다" 또는 "모르겠다" → 삭제.
- 답이 "구체적 위반 패턴 X가 재발한다" → 유지.

### Test 2: 비추론 테스트
- "이 규칙은 코드·문서·다른 상위 규칙에서 추론할 수 없는 정보인가?"
- 추론 가능 → 중복. 상위 규칙으로 위임 또는 삭제.
- 추론 불가능 → 유지.

### Test 3: 중복 테스트
- "이 규칙과 동일하거나 유사한 차단력을 가진 다른 규칙이 이미 있는가?"
- 있음 → 통합 또는 한쪽 삭제.
- 없음 → 유지.

### Test 4: 행동 변화 테스트
- "이 규칙이 있을 때와 없을 때, 에이전트 행동이 실제로 달라지는가?"
- 동일 → 토큰 낭비. 삭제.
- 다름 → 유지. (실제 차단되는 시나리오를 1줄로 본문에 명시)

### Test 5: 보편성 테스트
- "이 규칙이 단 1건의 사례에만 해당하지 않는가?"
- 1건 한정 → 보편 원리로 추상화하거나 삭제.
- 다수 사례에 적용 → 유지.

## 3. 토큰 = 비용

- 모든 규칙은 매 turn마다 컨텍스트에 로드된다. 추가는 영구 비용.
- 같은 차단력을 더 적은 토큰으로 얻을 수 있으면 그쪽이 default.
- 길이 < 명확성. 모호한 1줄보다 명확한 3줄이 낫지만, 동일 명확성이면 짧은 쪽.

## 4. 역사 박제 금지

- "왜 이 규칙이 생겼는가"의 역사 서술 금지. 보편 원리만.
- "예전에 X가 있었는데..." → 보편 원리로 표현하면 자연스럽게 차단된다.
- ADR·PR·CHANGELOG가 역사의 SSOT. 하네스는 **현재 적용되는 규칙의 SSOT**.

## 5. 규칙 작성 형식

- **명령형**: "~한다", "~하지 않는다". 권유형 ("~하면 좋다") 금지.
- **차단 조건과 행동**: "X 발견 시 → Y 수행" 또는 "X 금지". 모호한 형용사 ("적절히", "필요시") 금지.
- **예시는 최소화**: 보편 원리가 명확하면 예시 없이. 예시는 모호 해소가 필요할 때만 1-2개.

## 6. 추가·수정 워크플로

1. 변경안을 작성한다.
2. 5-Test 게이트를 자가 점검한다 (각 테스트의 답을 메모).
3. 통과 시 `/optimize-harness`를 호출하여 외부 검증.
4. 외부 게이트 통과 시 머지.

`/optimize-harness`는 5-Test를 외부 서브에이전트로 재검증하고, 추가로 다른 규칙과의 충돌·중복·포인터 단절을 검사한다.
